# Database

## Overview

The entire platform uses **Cloud Firestore** (Firebase) as its database. Firestore is a NoSQL document database organized into collections of documents. There is no relational schema — relationships are handled through **denormalized fields** (copying name/avatar at write time) and **UID references** (storing UIDs that the application resolves on read).

All services connect to Firestore using the **Firebase Admin SDK** with service account credentials from `backend/serviceAccountKey.json`.

---

## Collections

### `users`

Each document ID is the Firebase Authentication UID.

```
users/{uid}
├── uid: string                 — Firebase Auth UID
├── email: string
├── name: string
├── avatar: string              — URL (Cloudinary or DiceBear generated)
├── role: "viewer" | "contributor" | "admin"
├── emailVerified: boolean
├── createdAt: number           — ms epoch (Date.now())
└── updatedAt: number           — ms epoch
```

**Indexes:** Queried by `role` for admin dashboards.

---

### `projects`

Each document ID is auto-generated by Firestore.

```
projects/{projectId}
├── id: string                  — same as document ID
├── title: string
├── description: string
├── ownerId: string             — users/{uid} reference
├── ownerName: string           — denormalized
├── status: "pending" | "approved" | "rejected"
├── tags: string[]              — array of tag names
├── techStack: string[]         — e.g. ["React", "Python", "Firebase"]
├── contributors: string[]      — array of user UIDs
├── domain: string              — one of 9 predefined domains
├── isFeatured: boolean
├── isDeleted: boolean          — soft delete flag
├── commentCount: number        — denormalized from comments collection
├── assets: Asset[]             — embedded or subcollection (see below)
├── githubUrl: string
├── demoUrl: string
├── repoUrl: string
├── createdAt: number           — ms epoch
└── updatedAt: number           — ms epoch
```

**Ordering:** Listed with `orderBy("createdAt", "desc")` by default.

#### Embedded Asset Shape (within `projects`)

```
assets[i]
├── id: string
├── url: string              — Cloudinary secure_url
├── filename: string
├── contentType: string      — MIME type e.g. "image/jpeg"
└── size: number             — bytes
```

### `comments`

Top-level collection. Each document ID is auto-generated.

```
comments/{commentId}
├── id: string
├── projectId: string        — projects/{id} reference
├── userId: string           — users/{uid} reference
├── authorName: string       — denormalized at write time
├── authorAvatar: string     — denormalized at write time
├── text: string             — 1–5000 chars
├── likes: number
├── likedBy: string[]        — array of user UIDs
├── timestamp: string        — ISO 8601
├── edited: boolean
└── editedAt: string | null  — ISO 8601 or null
```

**Queried by:** `projectId == :id` filter, then sorted in memory.

---

### `tags`

```
tags/{tagId}
├── id: string
├── name: string             — display name e.g. "Machine Learning"
├── slug: string             — URL-safe e.g. "machine-learning"
└── usageCount: number       — not always kept in sync
```

---

### `events`

Managed by the Notification Service.

```
events/{eventId}
├── id: string
├── title: string
├── description: string
├── startDate: number        — ms epoch
├── endDate: number          — ms epoch
├── type: string             — e.g. "hackathon", "workshop", "talk"
├── createdAt: number
└── updatedAt: number
```

## Timestamp Handling

All services store timestamps as **millisecond epoch integers** (`Date.now()`). However, Firestore may return them as `Timestamp` objects with `_seconds` and `_nanoseconds` properties when read via the Admin SDK.

The gateway and services normalize using:
```js
function toTimestamp(value) {
  if (!value) return 0;
  if (typeof value === 'number') return value;
  if (value.toMillis) return value.toMillis();
  if (value._seconds) return value._seconds * 1000;
  return 0;
}
```

The frontend `parseDate()` helper additionally handles gRPC's `longs: String` serialization:
```js
function parseDate(date) {
  if (!date) return null;
  let parsed;
  if (date._seconds) {
    parsed = new Date(date._seconds * 1000);
  } else if (typeof date === 'string' && /^\d+$/.test(date)) {
    parsed = new Date(parseInt(date, 10));   // numeric string from gRPC
  } else {
    parsed = new Date(date);
  }
  return isNaN(parsed) ? null : parsed;
}
```

## Relationships

Firestore has no foreign key constraints. Relationships are modeled as:

| Relationship | Implementation |
|---|---|
| Project → Owner | `projects.ownerId` (UID string) |
| Project → Contributors | `projects.contributors` (array of UIDs) |
| Project → Tags | `projects.tags` (array of tag names, not IDs) |
| Comment → Project | `comments.projectId` |
| Comment → Author | `comments.userId` |

Resolution happens at read time in the gateway or frontend using parallel queries.

---

## No Migrations

Firestore is schema-less — fields can be added or removed without migrations. New fields added by code updates are simply absent on older documents (reads return `undefined`). All service code uses `|| defaultValue` guards for backward compatibility.

---

## Indexes

Firestore automatically indexes all top-level fields for simple equality queries. Composite indexes are required for queries that combine `where` and `orderBy`. Known required indices:

- `comments` collection: `projectId ASC, timestamp DESC` (for the fallback chronological reader)
- `projects` collection: `createdAt DESC` (auto-indexed, default listing order)

> ⚠️ If you see a Firestore error `"The query requires an index"`, follow the link in the error to create the required composite index in the Firebase Console.
